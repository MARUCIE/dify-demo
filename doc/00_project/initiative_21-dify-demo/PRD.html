1. 概述

PROJECT_DIR: /Users/mauricewen/Projects/21-dify-demo

1.1 产品名称

灵阙智能体平台 (LingQue Enterprise Agent Platform) -- Demo Edition

1.2 产品类型

企业级智能体平台 Demo 前端 -- 完整复刻 07-lingque-professional 的 B2B SaaS 架构，覆盖智能体管理、工作流编排、模板库、知识库、历史审计、平台设置六大模块。
v3.0 从单页 Demo 升级为完整平台展示。

1.3 背景

路桥集团业务场景特殊：

  • 偏远工地：项目在不开放的小地方（宜宾挂弓山等），基础设施有限
  • 大量手写单据：公务接待审批单、接待清单等多为手写填写，OCR 难度高
  • 批量审核需求：财务集中处理时，一次数十到数百笔报销单需同时审核
  • 合规要求严格：国企报销需经过多级审批，审核维度多、标准复杂

v3.0 扩展：客户不仅需要看到单个审核 Demo，更需要看到完整的企业智能体平台能力：

  • 多智能体管理与监控
  • 工作流可视化编排 (n8n 风格)
  • 知识库管理 (向量化索引)
  • 决策审计追踪 (合规要求)
  • 模型配置与安全管控

1.4 成功指标

  • 平台感：从「一个 Demo」变成「一个平台」，客户能看到完整的产品形态
  • 功能闭环：每个侧栏入口都有真实页面，不存在灰色/disabled 入口
  • 品牌一致：双色系贯穿全平台 (Sky Blue 平台 + Jade Teal 智能体)
  • 专业级 UX：frosted glass + 微交互 + 60fps 动效，匹配 B2B SaaS 头部产品

1.5 设计范式：Single-Screen Progressive Disclosure

整个 upload → processing → result 流程在 一个视口高度 内完成：

Phase 1 [上传态]: 全屏 dropzone（支持文件/文件夹拖拽）
    ↓ 动画收缩
Phase 2 [审核态]: 上传区缩为顶部条 → 工作流/批量进度占满主区域
    ↓ 动画收缩
Phase 3 [结果态]: 工作流缩为完成条 → 结果面板展开
    ↓ 点击展开
Phase 4 [详情态]: 单笔报销的详细审核报告（问题列表）

3. 核心功能

3.1 智能上传 (Smart Upload) -- v2.0 升级

功能描述：支持单文件、多文件、整个文件夹拖拽批量上传。

项目规格
上传方式Drag-and-drop zone (文件/文件夹) + click-to-browse
文件夹支持拖入文件夹自动递归扫描 PDF
文件类型仅 PDF (自动过滤非 PDF)
文件大小限制单文件 20MB
批量限制最多 100 文件/次
日期输入可选日期选择器，默认今天，应用于所有文件
校验文件类型检查、大小检查、重复检查
批量 UX文件列表表格（文件名/大小/页数/状态），带全选/删除操作
手写标识扫描检测手写比例，标记"手写为主"/"打印为主"

3.1.1 渐进式披露 -- 上传阶段

  • 初始态：全屏 dropzone，大面积拖拽区域，文字引导"拖拽文件或文件夹至此"
  • 有文件态：dropzone 缩小至 40% 高度，下方展示文件列表表格
  • 启动后：dropzone + 列表整体缩为顶部固定条（显示"X 个文件 · 审核中"）

3.2 工作流可视化 (10-step pipeline)

功能描述：以动画形式展示 10 步审核流程的执行过程。

10 个步骤：

步骤中文名称英文名称图标建议
1报销资料输入Document InputFileUp
2报销过期日判定Expiry CheckCalendar

4报销单据识别Receipt RecognitionScan
5提取接待类型信息Extract Reception InfoSearch
6判断接待类型Determine Reception TypeGitBranch

连接线动画：Steps 之间的连线在 active 状态时显示流动粒子效果。

布局：横向流水线布局 (desktop)，支持自动换行。

3.3 审核结果展示 (Result Display)

功能描述：展示结构化的审核结果。

结果结构：

招待类型：公务接待
审核建议：人工复核
问题列表：
  [!] 菜品清单缺失
  [!] 陪餐人员信息不完整
  [!] 报销日期超过有效期限
元素规格
汇总卡片招待类型 + 审核建议 (badge style) + 报销金额 + 审核耗时
子单据网格单据拆分识别: 6 种标准单据 (费用报销单/审批单/接待清单/情况说明/发票/菜品清单) 的 found/missing 状态 + 页码
问题列表严重程度编码项 (error=红, warning=琥珀, info=蓝)，支持详情展开
详情面板每项问题可展开查看详情
SubDocBar (批量看板)按文件展示子单据判定条：flex 比例色段 (green=通过, amber=警告, red=不通过+缺失)，72px 宽, 6px 高，带加粗计数标签
导出复制到剪贴板 (结构化文本)，PDF 导出 (计划中)

子单据类型 (单据拆分识别)：

#类型是否必需说明

3公务接待清单是来宾名单、人数、场所、行程

5电子发票是税务发票

建议颜色含义

4. 非功能性需求

4.1 性能

  • LCP (Largest Contentful Paint) < 2s
  • 工作流动画：60fps，无卡顿
  • 总包体积 < 500KB (gzipped)

4.2 可访问性

  • 全中文 UI (所有标签、提示、消息)
  • 核心流程支持键盘导航
  • 交互元素提供 Screen reader 标签

4.3 视觉设计

  • 专业企业美学 (国企风格)
  • 仅浅色主题 (frosted glass + 白色底色; 参见 D-011)
  • 双品牌色系：Sky Blue (#0284C7, 平台页面) + Jade Teal (#0D9488, 智能体页面)，中性灰色，红/绿/琥珀用于状态指示
  • 字体：系统中文字体 (PingFang SC / Microsoft YaHei fallback)
  • 微妙动效：不浮夸，但精致自信

4.4 响应式

  • 主要：桌面端 (1280px+)
  • 次要：平板横屏 (1024px+)
  • 不要求：移动端 (< 768px)

4.5 浏览器支持

  • Chrome 120+, Edge 120+, Safari 17+
  • Firefox 120+ (nice to have)

6. API 设计

6.1 模式

模式配置行为

6.2 接口

POST /api/audit (实际实现)

请求：multipart/form-data

  • file: PDF 二进制 (必需, 最大 20MB)
  • auditDate: "2025-07-09" (可选, 默认今天)
  • fileIndex: number (批量处理时使用, 0-based)

响应：SSE stream (text/event-stream)

event: step_start
data: {"step":1,"message":"正在接收报销资料..."}

event: step_done
data: {"step":1,"message":"报销资料接收完成","durationMs":800}

event: result
data: {"receptionType":"公务接待","suggestion":"人工复核","amount":470,"pageCount":5,"subDocuments":[...],"issues":[...]}

event: error
data: {"message":"Dify workflow failed (500): ..."}

event: done
data: {}

6.3 Dify API 集成

  • 接入点：https://agentdemo.hegui.cn/v1/workflows/run
  • 认证：Bearer token，存储在 DIFY_API_KEY
  • 输入映射：PDF 文件 → Dify 文件上传 → workflow 输入
  • 输出映射：Dify workflow nodes → 10 步进度事件
